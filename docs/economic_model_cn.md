## 经济模型分析

本文档基于合约实际实现，分析协议的经济模型、各角色的收益与动机，并评估潜在攻击面。

---

### 1. 资金流全景

```
  创世注入 + 读者打赏 + 污染罚没 + 分叉费
                    │
                    ▼
              ┌──────────┐  ←── Keeper 奖励（定额直扣，池不足则不发）
              │  奖金池   │  ←── 协议费（最高 10%，从 epochRelease 扣除）
              └─────┬────┘
                    │ Epoch 释放 = poolBalance × prizeReleaseRate / 10000
           ┌────────┼────────┐
           ▼        ▼        ▼
        创世者    作者奖金   投票者奖励
        分成      (均分)    (准确加权)

  未揭示投票质押 → 按比例分给已揭示投票者（不经奖金池）
```

**资金来源**

| 来源 | 流向 | 说明 |
|------|------|------|
| 创世注入 | 奖金池 | `createNovel()` 附带的 ETH |
| 读者打赏 | 奖金池 | `tipNovel()` 最低 0.001 ETH |
| 污染罚没 | 奖金池 | 被罚作者保证金的 50% |
| 分叉费 | 原小说奖金池 | 等于原小说的 `stakeAmount` |

**资金去向**

| 去向 | 触发时机 | 计算方式 |
|------|----------|----------|
| 协议金库 | Epoch 结算 | `totalRelease × protocolFeeRate / 10000` |
| 创世者分成 | Epoch 结算 | `totalRelease × G / (G + C)` |
| 作者奖金 | Epoch 结算 | `remaining × (10000 - voterRewardRate) / 10000`，按章节均分 |
| 投票者奖励 | Epoch 结算 | `remaining × voterRewardRate / 10000`，准确性加权 |
| Keeper 奖励 | 每次状态推进 | 固定金额，池余额不足则不发 |

---

### 2. Epoch 三层分配公式

每 Epoch 结算时释放奖金池余额的一部分：

```
totalRelease = poolBalance × prizeReleaseRate / 10000    (最高 50%)

① 协议费   = totalRelease × protocolFeeRate / 10000      (最高 10%，可为 0)
② 创世者   = (totalRelease - 协议费) × G / (G + C)
③ remaining = totalRelease - 协议费 - 创世者分成
④ 作者奖金  = remaining × (10000 - voterRewardRate) / 10000
⑤ 投票者池  = remaining - 作者奖金
```

**创世者分成衰减示例**（G=1）：

| Epoch | C (累计 Canon 章节) | 创世者占比 |
|-------|---------------------|-----------|
| 1 | 1 | 50% |
| 2 | 2 | 33% |
| 5 | 5 | 17% |
| 10 | 10 | 9% |
| 50 | 50 | 2% |

多章创世（G>1）会拉高初始占比，但衰减曲线相同。

### 3. 投票者收益模型

投票者有两层独立收益：

**第一层：未揭示质押再分配（每轮独立，不消耗奖金池）**

```
myShare = totalUnrevealedStake × myStake / totalRevealedStake
```

未 Reveal 的投票者质押被没收，按比例分给已 Reveal 的投票者。

**第二层：准确性奖励（Epoch 结算时发放）**

```
准确投票者权重 = stake × 3
其他投票者权重 = stake × 1

totalWeight = totalRevealedStake + totalAccurateStake × 2
myReward    = voterRewardPool × myWeight / totalWeight
```

"准确"定义：Round 投票中投了最终世界线 / Epoch 投票中投了 Canon。

关键设计：投错也有收益（1x 权重），只是比投准（3x 权重）少。这避免了"全有或全无"导致的参与意愿下降。

### 4. 各角色收益与动机

#### 4.1 创世者

| 收益 | 说明 |
|------|------|
| 持续分成 | `G/(G+C)` 衰减，但只要小说活跃就持续获得 |
| 创世 NFT | Canon 确立后铸造 ERC-721 |
| 分叉费 | 基于本小说分叉的新小说需缴纳分叉费 |

**动机**：创建高质量世界观吸引续写者和读者，维持小说活跃度以获得长期分成。

#### 4.2 续写作者

| 收益 | 说明 |
|------|------|
| 作者奖金 | Canon 章节按篇均分，多章多份 |
| 版权 NFT | ERC-721 + ERC-2981 版税（默认 5%）|
| 保证金返还 | 正常落选全额返还 |

| 成本 | 说明 |
|------|------|
| 保证金 | 每次提交质押 `stakeAmount` |
| Gas | 提交交易的 Gas 费 |
| 污染罚没 | 连续 M 轮排名底部 → 罚没 50% 保证金 |

**动机**：写好内容进入 Canon 获得 NFT 和奖金。污染机制惩罚持续低质量提交。

#### 4.3 投票者

| 收益 | 说明 |
|------|------|
| 质押返还 | Reveal 后全额返还 |
| 未揭示分配 | 瓜分忘记 Reveal 者的质押 |
| 准确性奖励 | 投中赢家获 3x 权重分配 |

| 成本 | 说明 |
|------|------|
| 质押 | Commit 时附带 ETH |
| 2× Gas | Commit + Reveal 两次交易 |
| 未 Reveal 罚没 | 忘记 Reveal 则质押被没收 |

**动机**：认真评审并投出最佳选择。Commit-Reveal 防止跟风，准确性乘数奖励独立判断。

#### 4.4 Keeper

| 收益 | 说明 |
|------|------|
| 固定奖励 | 每次成功调用状态推进函数获得定额奖励 |

| 成本 | 说明 |
|------|------|
| Gas | 调用交易的 Gas 费 |

**动机**：推进协议状态运转，获取小额确定性收益。Permissionless 设计确保任何人都可以参与。

#### 4.5 读者/打赏者

| 收益 | 说明 |
|------|------|
| 支持喜爱的故事 | 打赏直接进入奖金池，激励作者 |

### 5. 攻击面分析

#### 5.1 自写自投（Self-Vote）

**攻击方式**：攻击者自己提交章节，再用大量质押投票给自己。

**防御分析**：
- Stake-to-Vote 意味着攻击成本 = 质押资金的机会成本
- 攻击者只能提取奖金池已有资金，不能凭空创造价值
- 差内容无人打赏 → 奖金池枯竭 → 收益归零
- 其他投票者的存在分散了攻击者的控制力

**结论**：在奖金池较小的情况下，攻击成本（质押资金 + Gas）很可能超过收益。但当奖金池很大且参与者很少时，自写自投可能有利可图。这是 Stake-to-Vote 机制的固有局限——通过经济博弈而非技术手段抑制，属于可接受的风险。

#### 5.2 对冲投票（Hedge Voting）

**攻击方式**：攻击者向多个候选章节分散投票，确保无论谁赢都有收益。

**防御分析**：
- 3x 准确性乘数使集中投票的期望收益高于分散投票
- 假设两个候选各 50% 胜率：
  - 集中投票期望权重：`0.5 × 3S + 0.5 × 1S = 2S`
  - 对冲投票（各投一半）期望权重：`0.5S × 3 + 0.5S × 1 = 2S`
- 对冲投票在两候选等概率时期望收益相同，但当投票者有信息优势时，集中投票严格更优

**结论**：3x 乘数在信息不对称场景下有效抑制对冲。但在完全不确定的场景下，对冲和集中的期望收益相等——对冲不亏但也不赚额外收益，不构成攻击。

#### 5.3 女巫攻击（Sybil）

**攻击方式**：攻击者创建多个地址参与投票或提交。

**防御分析**：
- Stake-to-Vote：1 个地址投 1 ETH = 10 个地址各投 0.1 ETH，权重相同
- 投票权重完全取决于质押金额，拆分地址没有额外收益
- 续写提交需要保证金，多地址只是多付保证金，不增加胜率

**结论**：Sybil 攻击在纯 Stake-to-Vote 模型下无额外收益，设计上免疫。

#### 5.4 抢先交易（Front-Running）

**攻击方式**：在 Reveal 阶段观察他人的揭示交易，抢先提交自己的 Reveal。

**防御分析**：
- Commit-Reveal 两阶段设计：Commit 阶段提交 hash，Reveal 阶段揭示
- 攻击者无法在 Commit 阶段结束后修改投票
- 即使观察到他人 Reveal 的内容，自己的 Commit 已锁定
- Reveal 顺序不影响计票结果

**结论**：Commit-Reveal 机制完全防御了投票抢先交易。

#### 5.5 创世者灌水

**攻击方式**：创世者提交大量低质量创世章节以获取更高的初始分成比例。

**防御分析**：
- 创世章节受 `minChapterLength` 校验
- `genesisChapterCount ≤ worldLineCount`（合约限制）
- 章节越多 → 初始世界线越多 → Round 需要更多提交和投票 → 启动更慢
- 分成比例 `G/(G+C)` 在 C 增长后快速衰减

**结论**：合约强制 `genesisChapterCount ≤ worldLineCount`，且世界线数量本身受实际参与约束，灌水空间有限。

#### 5.6 投票者不 Reveal

**攻击方式**：投票者故意不 Reveal，试图阻碍计票。

**防御分析**：
- 未 Reveal 的质押被没收并分给已 Reveal 的投票者
- 不 Reveal 是纯损失行为，没有任何收益
- 计票只统计已 Reveal 的投票

**结论**：机制设计使"不 Reveal"成为严格劣势策略。

#### 5.7 Keeper 抢跑

**攻击方式**：多个 Keeper 竞争调用状态推进函数，获取奖励。

**防御分析**：
- 状态推进函数是 permissionless 的，先到先得
- 奖励固定且金额较小
- MEV 竞争可能导致 Gas 战争

**结论**：这是标准的 Keeper 竞争问题。奖励金额较小限制了 MEV 竞争的激烈程度。如果 Gas 战争成为问题，可通过调低 `keeperRewardAmount` 缓解。

#### 5.8 奖金池耗尽攻击

**攻击方式**：攻击者持续参与但从不打赏，通过提取奖励耗尽奖金池。

**防御分析**：
- 每 Epoch 只释放池余额的一定比例（最高 50%），余额指数衰减
- `balance × (1 - rate)^n` → 永远不会完全归零
- Keeper 奖励在池余额不足时不发放
- 没有外部资金注入的小说会自然衰亡（有意设计）

**结论**：指数衰减确保奖金池不会被一次性掏空。无打赏 → 奖励递减 → 参与者自然离开，是协议有意的"自然死亡"机制。

### 6. 已知局限与风险

#### 6.1 低参与度场景

当参与者极少时（如仅 1-2 个作者），自写自投的攻击成本降低。协议依赖 `roundMinSubmissions` 设置最低提交数量来缓解，但如果阈值设得太低，少数参与者可能控制投票结果。

**建议**：创世者应根据预期参与规模合理设置 `roundMinSubmissions`。

#### 6.2 Keeper 奖励静默失败

当奖金池余额不足以支付 Keeper 奖励时，`payKeeperReward()` 返回 false 但不 revert。状态推进正常执行，Keeper 无法获得奖励但也无法得知失败原因。

**影响**：低奖金池时 Keeper 可能停止服务，导致状态推进延迟。但由于函数是 permissionless 的，任何有动力推进状态的参与者都可以调用。

#### 6.3 投票者 Claim 中的转账顺序

`claimVotingReward()` 中先清零 `stakeAmount` 再转账。如果转账失败（如接收方是不接受 ETH 的合约），质押永久丢失。

**影响**：仅影响合约地址投票者。EOA 地址不受影响。这是 CEI 模式的标准权衡——防重入优先于转账失败恢复。

#### 6.4 污染记录持久性

作者的污染打击记录不会因停止参与而重置。如果作者在积累打击后暂停几轮再回来，旧记录仍然存在，但 `lastRecordedRound` 的间隔检查会将 `consecutiveStrikes` 重置为 1。

**影响**：设计合理，间隔参与会重置连续计数。

### 7. 设计评价

**优势**：
- 纯经济博弈，无治理代币，机制简洁
- Commit-Reveal 有效防止跟风和抢先交易
- 三层分配 + 衰减曲线平衡了各方利益
- Pull 模式领取避免了批量转账的 Gas 问题和 DoS 风险
- 指数衰减的奖金池释放保证了长期可持续性

**设计原则的体现**：
- **简单**：一个公式解决创世者分成（`G/(G+C)`），一个乘数解决准确性激励（3x），没有复杂的多轮博弈
- **可靠**：所有 ETH 转账使用 ReentrancyGuard + CEI 模式，边界条件（零余额、无投票者）显式处理
- **易升级**：UUPS Proxy + 50 slots gap，新功能（如 ReportRegistry、协议金库）作为独立合约或新参数添加，不改变已有存储布局
