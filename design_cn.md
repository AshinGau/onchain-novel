## 🔮 产品需求文档 (PRD)：去中心化协作式小说协议

### 1. 项目背景与愿景

随着生成式 AI 的爆发，大量小说和内容正由 AI Agent 或人类与 AI 协同创作。然而，AI 生成的随机性以及不同创作者的想法差异，导致故事情节质量参差不齐。

本项目旨在构建一个部署在以太坊（或 EVM 兼容的 L2）上的智能合约平台，通过 **"分支 → 共识 → 确权 → 激励"** 的闭环机制，将碎片化的创造力聚合为高质量的文学作品。平台允许全网用户和 AI Agent 共同参与小说的续写，通过读者社区的投票选出最优发展方向，通过奖金池激励创作者持续贡献，最终打造出由社区共建、版权链上确权的开源/共创文学作品。

---

### 2. 术语表

| 术语 | 定义 |
|------|------|
| **Novel（小说）** | 一个独立的协作创作项目，由创世者发起，拥有独立的合约实例或链上记录 |
| **Chapter（章节）** | 一次续写提交的内容单元 |
| **Branch（分支）** | 某个章节节点下的一条候选续写，类似 Git 的 branch |
| **World Line（世界线）** | 经过 Round 投票后保留的分支，被确认为"合理发展方向" |
| **Canon（正统主线）** | 经过 Epoch 投票后确立的唯一故事线，代表社区共识 |
| **Round（轮次）** | 一个"续写 → 投票 → 世界线收束"的完整周期 |
| **Epoch（纪元）** | 若干 Round 后的终极投票与结算周期，确立 Canon 并触发 NFT 铸造与奖金分配 |
| **Prize Pool（奖金池）** | 由创世者初始注入 + 读者打赏累积的 ETH 资金池，用于激励入选主线的创作者 |
| **Fork（分叉）** | 基于落选分支发起的全新独立小说 |

---

### 3. 核心创新点与价值主张

* **群体智慧碰撞**：打破传统单人网文作者的局限，发挥不同人类作者、不同 AI Agent 的创造能力和随机性，在协作与竞争中碰撞出意想不到的精彩剧情。
* **社区共识驱动内容进化**：引入读者投票机制作为"进化过滤器"，有效过滤低质量或逻辑崩坏的冗余情节，确保故事始终向最精彩的方向发展。
* **经济激励闭环**：通过创世奖金池 + 读者打赏 → Epoch 按贡献分配的机制，让优秀创作者获得实际经济回报，形成"好内容 → 更多读者 → 更多打赏 → 更多好内容"的正向飞轮。
* **链上确权与可组合版权**：利用区块链不可篡改的特性记录所有贡献者的版权。落选的"废案"也支持被分叉（Fork），溯源完整，结合版权 NFT 实现 Web3 创作者经济。

### 4. 核心功能需求

以下功能模块将主要通过智能合约实现交互与状态管理。

#### 模块一：项目发起与创世（Genesis）

* **创世发布**：任何链上地址（创作者）可发起一本新小说，提交小说大纲、世界观框架或前几章内容，并设定该小说的基本规则（以下参数在创世时确定，后续不可更改）：
  * 每次续写的最大字节长度（如 10,000 字节）
  * 每次续写的最小字节长度（如 500 字节）
  * Round 的时间窗口长度（如 7 天）
  * Round 触发投票所需的最低续写提交数量
  * 每轮 Round 保留的世界线数量 N（如 N=3）
  * 每个 Epoch 包含的 Round 数量 K（如 K=5）
  * 投票方式选择（代币加权 / 二次方投票 / Stake-to-Vote，详见模块三）
* **创世奖金池（可选）**：创世者在创建小说时，可以通过 `payable` 函数发送一笔 ETH 作为初始奖金池。该奖金池将在后续 Epoch 结算时分配给入选主线的创作者（详见模块七）。最低注入额度为 0 ETH（即可不注入）。
* **游戏开启**：创世内容上链后，智能合约自动开启创作游戏的主干线，进入第一个 Round 的续写阶段，允许全网用户参与续写。

#### 模块二：分支续写机制（Branching）

* **续写提交**：参与者可以在当前活跃的任意世界线的最新节点上提交续写内容。
* **质量门槛**：
  * **格式校验**：智能合约校验提交内容的字节长度是否在创世规则设定的 `[minLength, maxLength]` 范围内，不合规则拒绝。
  * **提交保证金（Stake）**：续写者需质押少量 ETH 作为保证金。保证金的处理规则如下：
    * **正常落选**：保证金**全额返还**——单次落选不罚没，鼓励创作者大胆尝试
    * **入选世界线**：保证金全额返还，并额外获得奖金池分配的奖励
    * **罚没条件（罚没 50%，其余返还，罚没部分划入奖金池）**：
      * (a) **字数不达标**：提交内容的链下实际字数低于创世规则的 `minLength`（链上通过社区举报 + 仲裁验证）
      * (b) **持续污染**：该作者在最近连续 M 次（如 3 次）Round 投票中均排名倒数（如后 20%），表明持续产出低质量内容
  * **内容哈希校验**：提交时附带链下内容的 Content Hash（CID），合约记录但不验证内容本身——内容验证依赖社区投票的自然筛选。
* **平行故事线生成**：每次合规的续写提交，都在链上数据结构中创建一条独立分支（Branch），记录父节点 ID、分支 ID、作者地址和 Content Hash。

#### 模块三：轮次投票与世界线收束（Round Voting）

* **Round 触发条件**（满足任一条件即触发投票阶段）：
  * 当前 Round 的时间窗口到期（如创世规则设定的 7 天）
  * 当前活跃世界线上累计的分支提交数量达到设定阈值
* **投票权重机制**（创世时选择以下方案之一）：
  * **(a) 代币加权**：1 token = 1 vote，简单直接，适合有原生治理代币的项目
  * **(b) 二次方投票（Quadratic Voting）**：投给同一分支的票数越多，单票成本越高（`cost = votes²`），平衡大户与小户的影响力
  * **(c) Stake-to-Vote**：质押 ETH 投票，Round 结束后返还。质押量 = 投票权重。激励投票者认真对待
* **Commit-Reveal 两阶段投票**（防止跟风和贿赂攻击）：
  1. **Commit 阶段**（如 3 天）：投票者提交加密的投票（`hash(branchId + salt)`），投票内容对外不可见
  2. **Reveal 阶段**（如 2 天）：投票者揭示原始投票和 salt，合约验证哈希一致性后计票。超时未 Reveal 的投票作废，质押物不返还
* **投票激励**：投票本身需消耗少量代币或 Gas。投票与最终多数结果一致的投票者，可获得少量奖金池奖励（Schelling Point 激励），鼓励独立思考而非随波逐流。
* **世界线收束**：投票结束后，得票数最高的 N 条续写被合约确认为"世界线"，故事基于这 N 条世界线继续向下发展，开启新的 Round。活跃世界线总数始终 ≤ N。

#### 模块四：纪元结算与主线确立（Epoch Consensus）

* **Epoch 自动触发条件**：当前小说经过 K 个 Round（K 在创世时设定）后，合约自动进入 Epoch 投票阶段。无需管理员手动触发。
* **社区提前触发**：允许社区通过链上提案 + 投票的方式提前触发 Epoch（需达到最低投票率门槛，如 30% 的活跃投票者同意）。
* **唯一主线确立**：在 Epoch 投票中，社区从当前保留的 N 个世界线中，票选出唯一一条最精彩的故事线作为本纪元的正统主线（Canon）。投票机制与 Round 一致（Commit-Reveal + 权重）。
* **Epoch 结算**：主线确立后自动触发以下结算动作：
  1. 为主线章节的作者铸造版权 NFT（详见模块五）
  2. 分配奖金池中的 Epoch 份额给主线作者（详见模块七）
  3. 返还所有正常参与者的续写保证金；对触发罚没条件的作者（字数不达标 / 持续污染）执行 50% 罚没
  4. 重置状态，以确立的 Canon 为基础开启新一轮 Round

#### 模块五：版权确权与 NFT 铸造

* **创作者结算**：在每个 Epoch 结束并确立 Canon 后，智能合约自动触发结算。
* **版权 NFT 铸造**：为入选 Canon 的每个章节的原作者，自动铸造一枚"章节版权证明 NFT"（ERC-721）。该 NFT 包含以下元数据：
  * 小说 ID、章节 ID、Epoch 编号
  * 作者钱包地址
  * 内容 CID（IPFS/Arweave 链接）
  * 铸造时间戳
* **NFT 定位**：当前定位为**链上版权证明 + 荣誉凭证**。持有该 NFT 可证明链上创作者身份与贡献记录。未来可视社区需求接入 EIP-2981 版税标准，实现 NFT 二级交易时的自动版税分配。
* **Epoch 奖金联动**：NFT 铸造与奖金分配同步执行——铸造 NFT 的同一交易中记录该作者可领取的奖金数额（详见模块七）。

#### 模块六：废案利用与分叉（Forking）

* **落选数据保护**：未被选入主线的其他故事分支不会被删除，其完整记录（作者、内容哈希、时间戳、得票数）永久保留在链上。
* **衍生作品分叉**：如果部分读者对某个落选分支非常感兴趣，合约允许以该节点为基础创建一本**全新的独立小说**。Fork 遵循以下规则：
  * Fork 创建一个完全独立的小说实例（新的 Novel ID、独立的奖金池与 Round/Epoch 周期）
  * Fork 来源信息在链上不可篡改地记录（`forkFrom: {originalNovelId, branchId}`），保持完整溯源
  * 原小说的数据和奖金池不受影响——Fork 不会分割原小说的任何资产
  * 原作者在分叉宇宙中的章节版权仍受智能合约保护

#### 模块七：奖金池与激励机制（Prize Pool & Incentives） 🆕

##### 7.1 奖金池资金来源

| 资金来源 | 说明 |
|----------|------|
| **创世注入** | 创世者发起小说时通过 `payable` 发送的初始 ETH |
| **读者打赏** | 读者调用 `tipNovel()` 向小说打赏的 ETH（详见模块八） |
| **违规保证金罚没** | 字数不达标或持续污染的续写者被罚没的 50% 保证金 |
| **过期投票质押没收** | Commit-Reveal 中未按时 Reveal 的投票者质押物 |

##### 7.2 奖金分配规则

* **分配时机**：每个 Epoch 结算时分配一次。
* **释放比例**：每个 Epoch 释放当前奖金池余额的固定百分比（如 30%），而非全额释放，确保后续 Epoch 仍有激励，实现长期可持续。
* **Release Schedule 公式**：`epochReward = poolBalance × releaseRate`（`releaseRate` 在创世时设定，如 30%）
* **分配逻辑**：释放的奖金按入选 Canon 的章节数量均分给对应作者（每章等额）。若某个作者贡献了多个章节，则获得多份。
* **领取方式**：Pull 模式——作者主动调用 `claimReward()` 提取已分配的奖金。不使用 Push 模式，避免因接收方合约拒绝转账导致 DoS。
* **未领取奖金处理**：已分配但长期未领取的奖金保持可领取状态，不设过期时间。

##### 7.4 参与者激励一览

| 角色 | 激励 | 代价 |
|------|------|------|
| **创世者** | 发起 IP、吸引社区、获得创世章节的 NFT 与奖金 | 注入初始奖金池（可选） |
| **续写创作者** | 入选 Canon → NFT + 奖金池分成 + 保证金返还；正常落选也全额返还保证金 | 续写保证金质押 + Gas 费（违规时罚没 50%） |
| **投票读者** | 与多数一致 → Schelling 奖励 | 投票质押 + Gas 费 |
| **打赏读者** | 支持喜爱的故事、可获赞助者荣誉 | 打赏金额 + Gas 费 |

#### 模块八：读者打赏（Reader Tipping） 🆕

* **打赏入口**：读者调用 `tipNovel(novelId)` 的 `payable` 函数，向指定小说打赏 ETH。
* **打赏对象**：打赏进入**小说级别的公共奖金池**，不直接打赏个人作者，避免激励失衡和投票博弈。
* **最低金额**：设定最低打赏金额（如 0.001 ETH），防止粉尘攻击和无意义交易。
* **打赏记录**：链上发出 `TipReceived(novelId, tipper, amount, timestamp)` 事件，便于前端展示和链下索引。
* **打赏者权益（可选增强，未来版本考虑）**：
  * 打赏累计达到一定阈值的读者，可获投票权重加成（如 1.2x）
  * 可铸造专属"赞助者 NFT 徽章"作为荣誉展示
* **汇入奖金池**：打赏金额实时累加到对应小说的 `poolBalance`，在下一个 Epoch 结算时统一参与分配。

### 5. 技术架构与存储方案

> **💡 关键提示**：以太坊主网的存储成本极高（Gas 费昂贵），绝对不适合直接存储小说文本。

#### 5.1 链上数据（智能合约）

仅存储轻量级元数据和控制逻辑：

* **小说元数据**：小说 ID、创世者地址、创世规则参数、Fork 来源信息
* **章节元数据**：章节 ID、作者钱包地址、父节点 ID、子节点 ID 列表、Content Hash（CID）
* **状态机**：Round / Epoch 状态（续写中 → 投票中 → 结算中）、计时器、计票器
* **经济数据**：奖金池余额、各地址的保证金锁定、已分配未领取的奖金数额
* **投票数据**：Commit 哈希记录、Reveal 后的明文票数

#### 5.2 链下数据（Web3 存储介质）

* 采用 **IPFS** 或 **Arweave**（永久存储）存储大段小说文本
* 用户前端将文本上传至 IPFS/Arweave，获取唯一 CID
* 随后在以太坊钱包中发起交易，将 CID 写入合约。既保证文本不可篡改，又极大降低 Gas 费用

#### 5.3 部署链选择

| 选项 | 优点 | 缺点 |
|------|------|------|
| **以太坊 L1** | 安全性最高、去中心化程度最强 | Gas 费极高，每次续写/投票可能花费数十美元 |
| **L2（如 Base / Arbitrum）** | Gas 低 100-1000 倍、用户准入门槛低 | 需信任 L2 运营者（安全性依赖 L1 最终一致性） |
| **应用链 / L3** | 完全自定义、Gas 可趋近于零 | 基础设施不成熟、用户迁移成本高 |

> **建议**：优先考虑 **EVM 兼容的 L2**（如 Base），在保障安全性的前提下大幅降低用户参与成本，有利于社区冷启动。

### 6. 治理与安全

#### 6.1 合约安全

* **重入攻击防护**：所有涉及 ETH 转移的函数（`claimReward`, `withdrawStake`）必须使用 `ReentrancyGuard`
* **Checks-Effects-Interactions 模式**：先更新内部状态，再执行外部调用
* **整数安全**：使用 Solidity ≥ 0.8.x，内置溢出保护
* **访问控制**：关键管理函数使用 `Ownable` 或 `AccessControl`
* **时间锁（Timelock）**：大额参数修改（如 `releaseRate` 调整）需经过时间锁延迟执行

#### 6.2 紧急治理

* **紧急暂停**：合约实现 `Pausable` 机制，多签管理员（如 3/5 多签）可在漏洞发现时紧急暂停续写、投票和提款
* **升级策略**：使用 UUPS Proxy 模式部署，允许在发现严重 Bug 时升级合约逻辑，升级操作需多签 + 时间锁双重保护

#### 6.3 内容审核说明

* 协议层面**不做内容审查**——这与去中心化理念一致
* 内容审核由前端/客户端自行决定（如过滤列表、社区举报）
* 合约只保证经济机制的正确执行和版权记录的不可篡改

### 7. 开放问题与未来扩展

以下事项需在实现阶段进一步细化：

1. **代币设计**：是否发行协议原生 ERC-20 治理代币？还是直接使用 ETH 作为唯一资产？
2. **投票权重具体参数**：二次方投票的基础单价、Stake-to-Vote 的最低质押量等需经济学仿真后确定
3. **保证金比例**：续写保证金金额、落选罚没比例（50%？30%？）需在 Testnet 上观察社区反馈后调整
4. **Epoch 释放率**：30% 是初始建议值，理想值需根据奖金池规模和参与人数动态调整
5. **跨小说互操作**：不同小说之间的角色/世界观能否交叉引用？这是高级特性，留待 V2
6. **AI Agent 准入**：是否对 AI Agent 的参与做特殊限制或标识？
7. **版税分配（V2）**：未来可接入 EIP-2981，实现版权 NFT 二级交易时的自动版税分配
