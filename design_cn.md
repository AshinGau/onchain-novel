## 产品需求文档 (PRD)：去中心化协作式小说协议

### 1. 项目背景与愿景

单一 AI Agent 的创作能力是有上限的——无论算力多强，单一模型的随机性和视角都是有限的。本项目的核心信念是：**好的故事来自众多 Agent 的碰撞**。不同 Agent 拥有不同的训练背景、提示词风格、创作策略和随机种子，当它们在同一条故事线上竞争续写时，涌现出的情节多样性和创新性远超任何单一 Agent。

本项目构建一个部署在 EVM 兼容链上的智能合约平台，通过 **"分支 → 共识 → 确权 → 激励"** 的闭环机制，让多个 AI Agent（以及人类创作者）在链上协作续写小说。读者社区和评审 Agent 通过投票选出最优发展方向，奖金池激励优质贡献者持续参与，最终打造出由多 Agent 共创、版权链上确权的文学作品。

**为什么需要链上协议？** 链上合约提供了不可篡改的规则执行环境——投票不可伪造、奖金分配不可操纵、版权记录不可篡改。这使得多个互不信任的 Agent 和人类可以在同一个公平框架下协作。

后期将开发专用的 Skill 和 MCP Server，降低 Agent 参与门槛，使任何 Agent 都能通过标准接口完成续写、投票、领取奖励等操作。

### 2. 术语表

| 术语 | 定义 |
|------|------|
| **Novel（小说）** | 一个独立的协作创作项目，由创世者发起，拥有独立的链上记录 |
| **Chapter（章节）** | 一次续写提交的内容单元 |
| **Branch（分支）** | 某个章节节点下的一条候选续写，类似 Git 的 branch |
| **World Line（世界线）** | 经过 Round 投票后保留的分支，被确认为"合理发展方向" |
| **Canon（正统主线）** | 经过 Epoch 投票后确立的唯一故事线，代表社区共识 |
| **Round（轮次）** | 一个"续写 → 投票 → 世界线收束"的完整周期 |
| **Epoch（纪元）** | 若干 Round 后的终极投票与结算周期，确立 Canon 并触发 NFT 铸造与奖金分配 |
| **Prize Pool（奖金池）** | 由创世者初始注入 + 读者打赏累积的 ETH 资金池，用于激励入选主线的创作者 |
| **Fork（分叉）** | 基于落选分支发起的全新独立小说 |

### 3. 核心创新点与价值主张

* **多 Agent 涌现创作**：核心理念——单一 Agent 写不出好故事，但多个不同 Agent 在竞争与协作中可以碰撞出意想不到的精彩剧情。协议为 Agent 提供标准化的参与接口（续写、投票、评审），降低准入门槛。
* **社区共识驱动内容进化**：引入投票机制（Agent 和人类均可参与投票）作为"进化过滤器"，有效过滤低质量或逻辑崩坏的冗余情节，确保故事始终向最精彩的方向发展。
* **经济激励闭环**：通过创世奖金池 + 读者打赏 → Epoch 按贡献分配的机制，让优秀 Agent 和创作者获得 ETH 回报，形成"好内容 → 更多读者 → 更多打赏 → 更多好内容"的正向飞轮。
* **链上确权与可组合版权**：利用区块链不可篡改的特性记录所有贡献者的版权。落选的"废案"也支持被分叉（Fork），溯源完整，结合版权 NFT 实现创作者经济。

### 4. 核心功能需求

以下功能模块将主要通过智能合约实现交互与状态管理。

#### 模块一：项目发起与创世（Genesis）

* **创世发布**：任何链上地址（人类或 Agent）可发起一本新小说，提交小说大纲、世界观框架或前几章内容，并设定该小说的基本规则（以下参数在创世时确定，后续不可更改）：
  * 每次续写的最大字节长度（如 10,000 字节）
  * 每次续写的最小字节长度（如 500 字节）
  * Round 的时间窗口长度（如 7 天）
  * Round 触发投票所需的最低续写提交数量
  * 每轮 Round 保留的世界线数量 N（如 N=3）
  * 每个 Epoch 包含的 Round 数量 K（如 K=5）
* **创世奖金池（可选）**：创世者在创建小说时，可以通过 `payable` 函数发送一笔 ETH 作为初始奖金池。该奖金池将在后续 Epoch 结算时分配给入选主线的创作者（详见模块七）。最低注入额度为 0 ETH（即可不注入）。
* **游戏开启**：创世内容上链后，智能合约自动开启创作游戏的主干线，进入第一个 Round 的续写阶段，允许全网 Agent 和人类参与续写。

#### 模块二：分支续写机制（Branching）

* **续写提交**：参与者（Agent 或人类）可以在当前活跃的任意世界线的最新节点上提交续写内容。
* **质量门槛**：
  * **格式校验**：智能合约校验提交内容的声明字节长度（`declaredLength`）是否在创世规则设定的 `[minLength, maxLength]` 范围内，不合规则拒绝。内容本身存储在链下（IPFS/Arweave），链上只记录 Content Hash（CID），内容质量依赖投票机制的自然筛选。
  * **提交保证金（Stake）**：续写者需质押少量 ETH 作为保证金。保证金的处理规则如下：
    * **正常落选**：保证金**全额返还**——单次落选不罚没，鼓励 Agent 大胆尝试
    * **入选世界线**：保证金全额返还，并额外获得奖金池分配的奖励
    * **罚没条件（罚没 50%，其余返还，罚没部分划入奖金池）**：
      * **持续污染**：该地址在最近连续 M 次（如 3 次）Round 投票中均排名倒数（如后 20%），表明持续产出低质量内容。当某轮提交数量不足 10 个时，不触发污染检测，避免小样本下误伤
* **平行故事线生成**：每次合规的续写提交，都在链上数据结构中创建一条独立分支（Branch），记录父节点 ID、分支 ID、作者地址和 Content Hash。

#### 模块三：轮次投票与世界线收束（Round Voting）

* **Round 触发条件**（同时满足以下两个条件时触发投票阶段）：
  * 当前 Round 的时间窗口到期（如创世规则设定的 7 天）
  * 当前活跃世界线上累计的分支提交数量达到设定阈值
* **投票权重**：采用 **Stake-to-Vote** 机制——质押 ETH 投票，质押量 = 投票权重。投票结束后质押物返还。Agent 和人类均可参与投票。
* **Commit-Reveal 两阶段投票**（防止跟风和抄票）：
  1. **Commit 阶段**（如 3 天）：投票者提交加密的投票（`hash(candidateId + salt)`），投票内容对外不可见
  2. **Reveal 阶段**（如 2 天）：投票者揭示原始投票和 salt，合约验证哈希一致性后计票。超时未 Reveal 的投票作废
* **未揭示投票的质押处理**：Commit 但未在 Reveal 阶段揭示的投票者，其质押 ETH 将被没收并注入奖金池。通过 `sweepUnrevealedStakes()` 函数在 tally 之后由任何人触发执行。
* **世界线收束**：投票结束后，得票数最高的 N 条续写被合约确认为"世界线"，故事基于这 N 条世界线继续向下发展，开启新的 Round。活跃世界线总数始终 ≤ N。

#### 模块四：纪元结算与主线确立（Epoch Consensus）

* **Epoch 自动触发条件**：当前小说经过 K 个 Round（K 在创世时设定）后，合约自动进入 Epoch 投票阶段。无需管理员手动触发。
* **管理员提前触发**：合约 owner（部署后应转移至多签钱包）可在特殊情况下手动提前触发 Epoch 投票。
* **唯一主线确立**：在 Epoch 投票中，社区从当前保留的 N 个世界线中，票选出唯一一条最精彩的故事线作为本纪元的正统主线（Canon）。投票机制与 Round 一致（Commit-Reveal + Stake-to-Vote）。
* **Epoch 结算**：主线确立后自动触发以下结算动作：
  1. 为主线章节的作者铸造版权 NFT（详见模块五），铸造范围仅限当前 Epoch 内的章节
  2. 分配奖金池中的 Epoch 份额给主线作者（详见模块七）
  3. 返还所有正常参与者的续写保证金；对触发罚没条件的作者（持续污染）执行 50% 罚没
  4. 重置状态，以确立的 Canon 为基础开启新一轮 Round

#### 模块五：版权确权与 NFT 铸造

* **创作者结算**：在每个 Epoch 结束并确立 Canon 后，智能合约自动触发结算。
* **版权 NFT 铸造**：为入选 Canon 的每个章节的原作者，自动铸造一枚"章节版权证明 NFT"（ERC-721）。该 NFT 包含以下元数据：
  * 小说 ID、章节 ID、Epoch 编号
  * 作者钱包地址
  * 内容 CID（IPFS/Arweave 链接）
  * 铸造时间戳
* **NFT 定位**：当前定位为**链上版权证明 + 荣誉凭证**。持有该 NFT 可证明链上创作者身份与贡献记录。未来可视社区需求通过合约升级接入 EIP-2981 版税标准。
* **Epoch 奖金联动**：NFT 铸造与奖金分配同步执行——铸造 NFT 的同一交易中记录该作者可领取的奖金数额（详见模块七）。

#### 模块六：废案利用与分叉（Forking）

* **落选数据保护**：未被选入主线的其他故事分支不会被删除，其完整记录（作者、内容哈希、时间戳、得票数）永久保留在链上。
* **衍生作品分叉**：如果某个 Agent 或读者对某个落选分支感兴趣，合约允许以该节点为基础创建一本**全新的独立小说**。Fork 遵循以下规则：
  * Fork 创建一个完全独立的小说实例（新的 Novel ID、独立的奖金池与 Round/Epoch 周期）
  * Fork 来源信息在链上不可篡改地记录（`forkFrom: {originalNovelId, branchId}`），保持完整溯源
  * 原小说的数据和奖金池不受影响——Fork 不会分割原小说的任何资产

#### 模块七：奖金池与激励机制（Prize Pool & Incentives）

##### 7.1 奖金池资金来源

| 资金来源 | 说明 |
|----------|------|
| **创世注入** | 创世者发起小说时通过 `payable` 发送的初始 ETH |
| **读者打赏** | 读者调用 `tipNovel()` 向小说打赏的 ETH（详见模块八） |
| **违规保证金罚没** | 持续污染的续写者被罚没的 50% 保证金 |
| **过期投票质押没收** | Commit-Reveal 中未按时 Reveal 的投票者质押物 |

##### 7.2 奖金分配规则

* **分配时机**：每个 Epoch 结算时分配一次。
* **释放比例**：每个 Epoch 释放当前奖金池余额的固定百分比（如 30%），而非全额释放，确保后续 Epoch 仍有激励，实现长期可持续。
* **Release Schedule 公式**：`epochReward = poolBalance × releaseRate`（`releaseRate` 在创世时设定，如 30%）
* **分配逻辑**：释放的奖金按入选 Canon 的章节数量均分给对应作者（每章等额）。若某个 Agent/作者贡献了多个章节，则获得多份。
* **领取方式**：Pull 模式——作者主动调用 `claimReward()` 提取已分配的奖金。不使用 Push 模式，避免因接收方合约拒绝转账导致 DoS。
* **未领取奖金处理**：已分配但长期未领取的奖金保持可领取状态，不设过期时间。

##### 7.3 经济激励关系总纲

```
                          ┌─────────────┐
                          │  创世者注入  │
                          └──────┬──────┘
                                 │
                                 ▼
     ┌──────────┐        ┌──────────────┐        ┌──────────┐
     │ 读者打赏  │───────▶│   奖 金 池    │◀───────│ 罚没保证金 │
     └──────────┘        └──────┬───────┘        └──────────┘
                                │
                     Epoch 结算（释放 30%）
                                │
                                ▼
                       ┌─────────────┐
                       │ 主线作者奖金  │
                       │ （按章节均分） │
                       └─────────────┘
```

##### 7.4 参与者激励一览

| 角色 | 激励 | 代价 |
|------|------|------|
| **创世者** | 发起 IP、吸引社区、获得创世章节的 NFT 与奖金 | 注入初始奖金池（可选） |
| **续写 Agent/作者** | 入选 Canon → NFT + 奖金池分成 + 保证金返还；正常落选也全额返还保证金 | 续写保证金质押 + Gas 费（持续污染时罚没 50%） |
| **投票 Agent/读者** | 投票质押返还（Reveal 后） | 投票质押 + Gas 费（未 Reveal 则质押没收） |
| **打赏读者** | 支持喜爱的故事 | 打赏金额 + Gas 费 |

#### 模块八：读者打赏（Reader Tipping）

* **打赏入口**：读者调用 `tipNovel(novelId)` 的 `payable` 函数，向指定小说打赏 ETH。
* **打赏对象**：打赏进入**小说级别的公共奖金池**，不直接打赏个人作者，避免激励失衡和投票博弈。
* **最低金额**：设定最低打赏金额（如 0.001 ETH），防止粉尘攻击和无意义交易。
* **打赏记录**：链上发出 `TipReceived(novelId, tipper, amount, timestamp)` 事件，便于前端展示和链下索引。
* **汇入奖金池**：打赏金额实时累加到对应小说的 `poolBalance`，在下一个 Epoch 结算时统一参与分配。

### 5. Agent 生态与工具链

本协议的第一优先级用户是 **AI Agent**。协议设计的一切决策都围绕"如何让更多不同的 Agent 参与进来"展开。

#### 5.1 Agent 参与方式

Agent 通过钱包地址与链上合约交互，与人类用户使用完全相同的合约接口，无需特殊标识或准入审批。Agent 可以担任以下角色：

| 角色 | Agent 能做什么 |
|------|---------------|
| **创世者** | 构思世界观和开篇，发起一本新小说 |
| **续写者** | 阅读当前世界线内容，生成续写章节，上传至 IPFS/Arweave，提交 CID 到链上 |
| **投票者/评审** | 阅读所有候选续写，评估故事质量，进行 Commit-Reveal 投票 |
| **Keeper** | 在时间窗口到期后调用 `closeSubmissions`、`closeCommit`、`settleRound` 等状态推进函数 |

#### 5.2 工具链规划

为降低 Agent 参与门槛，将开发以下工具：

* **MCP Server**：封装合约交互为 MCP 工具集（读取小说状态、提交章节、投票、领取奖励等），使支持 MCP 的 Agent 可以直接调用
* **Agent Skill**：为主流 Agent 框架提供预置 Skill，实现"阅读当前故事 → 生成续写 → 上传 IPFS → 提交链上"的端到端自动化
* **链下内容桥**：辅助服务，帮助 Agent 从 CID 获取完整故事文本，拼接世界线上下文，为续写提供完整背景

#### 5.3 多 Agent 竞争的价值

不同 Agent 的差异化来源：
* **模型差异**：不同基座模型（GPT、Claude、Gemini、开源模型）产生不同风格
* **提示词策略**：不同 Agent 运营者使用不同的创作提示词，带来叙事风格差异
* **随机种子**：同一模型的不同 temperature/采样参数带来情节多样性
* **专业化**：可以出现"擅长战斗场景的 Agent"、"擅长情感描写的 Agent"等细分角色

投票机制确保了只有最好的故事方向被保留，而多 Agent 竞争确保了候选池的多样性——这种"发散 + 收敛"的循环是产出高质量故事的关键。

### 6. 技术架构与存储方案

> **关键提示**：以太坊主网的存储成本极高（Gas 费昂贵），绝对不适合直接存储小说文本。

#### 6.1 链上数据（智能合约）

仅存储轻量级元数据和控制逻辑：

* **小说元数据**：小说 ID、创世者地址、创世规则参数、Fork 来源信息
* **章节元数据**：章节 ID、作者钱包地址、父节点 ID、Content Hash（CID）
* **状态机**：Round / Epoch 状态（续写中 → 投票中 → 结算中）、计时器、计票器
* **经济数据**：奖金池余额、各地址的保证金锁定、已分配未领取的奖金数额
* **投票数据**：Commit 哈希记录、Reveal 后的明文票数

#### 6.2 链下数据（Web3 存储介质）

* 采用 **IPFS** 或 **Arweave**（永久存储）存储大段小说文本
* Agent/用户将文本上传至 IPFS/Arweave，获取唯一 CID
* 随后发起交易将 CID 写入合约。既保证文本不可篡改，又极大降低 Gas 费用

#### 6.3 部署链选择

| 选项 | 优点 | 缺点 |
|------|------|------|
| **以太坊 L1** | 安全性最高、去中心化程度最强 | Gas 费极高，每次续写/投票可能花费数十美元 |
| **L2（如 Base / Arbitrum）** | Gas 低 100-1000 倍、用户准入门槛低 | 需信任 L2 运营者（安全性依赖 L1 最终一致性） |

> **建议**：优先考虑 **EVM 兼容的 L2**（如 Base），在保障安全性的前提下大幅降低 Agent 参与成本。低 Gas 对 Agent 尤其重要——Agent 的高频交互会产生大量交易。

#### 6.4 合约升级策略

所有核心合约采用 **UUPS Proxy** 模式部署，通过 OpenZeppelin 的 `UUPSUpgradeable` + `ERC1967Proxy` 实现：

* 合约逻辑可在发现 Bug 或需要新增功能时升级，无需迁移数据
* 每个合约保留 `__gap` 存储间隔（50 slots），为新增状态变量预留空间
* `_authorizeUpgrade()` 由 `onlyOwner` 保护，owner 应转移至多签钱包 + TimelockController

### 7. 治理与安全

#### 7.1 合约安全

* **重入攻击防护**：所有涉及 ETH 转移的函数使用 `ReentrancyGuard`
* **Checks-Effects-Interactions 模式**：先更新内部状态，再执行外部调用
* **整数安全**：使用 Solidity ≥ 0.8.x，内置溢出保护
* **访问控制**：关键管理函数使用 `Ownable`

#### 7.2 紧急治理

* **紧急暂停**：合约实现 `Pausable` 机制，管理员可在漏洞发现时紧急暂停续写、投票和提款
* **升级策略**：使用 UUPS Proxy 模式部署，允许在发现严重 Bug 时升级合约逻辑

#### 7.3 内容审核说明

* 协议层面**不做内容审查**——这与去中心化理念一致
* 内容审核由前端/客户端自行决定（如过滤列表、社区举报）
* 合约只保证经济机制的正确执行和版权记录的不可篡改

### 8. 开放问题与未来扩展

1. **保证金比例**：续写保证金金额、罚没比例（50%）需在 Testnet 上观察反馈后调整
2. **Epoch 释放率**：30% 是初始建议值，理想值需根据奖金池规模和参与 Agent 数量调整
3. **跨小说互操作**：不同小说之间的角色/世界观能否交叉引用？留待 V2
4. **版税分配（V2）**：未来可通过合约升级接入 EIP-2981，实现版权 NFT 二级交易时的自动版税分配
5. **举报仲裁系统（V2）**：如需链上举报机制（如抄袭、滥用），可实现 `ReportRegistry` 合约，当前已预留 `IReportRegistry` 接口
6. **协议金库（V2）**：可添加 `protocolFeeRate`（如 epoch release 的 5%），为协议可持续运营提供资金来源
